# Petstore API тестирование блока /pet

Тестирование публичного API [Petstore v2](https://petstore.swagger.io), блок `/pet`

## Инструменты

- [Insomnia](https://insomnia.rest)  выполнение запросов и проверка ответов

## Структура репозитория

```
Petstore-API-Pet-Testing/
├── PET_Store_Pet.yaml   — коллекция запросов Insomnia
├── pet_checklist.md — чеклист тестирования блока /pet 
└── README.md
```

## Как импортировать коллекцию

1. Открыть Insomnia
2. Import → выбрать `collections/PET_Store_Pet.yaml`

## Найденные баги

### POST /pet

1. Генерирует одинаковые `id` для разных объектов (если не передавать `id`), из-за чего ранее созданные сущности могут быть потеряны или недоступны  

2. Позволяет создавать сущности без обязательных полей (`name`, `photoUrls`), что приводит к неконсистентности данных  

3. Не валидирует содержимое поля `photoUrls` (принимает любые ссылки, не обязательно на изображения)  

4. Принимает недопустимые значения поля `status`, не входящие в список (`available`, `pending`, `sold`)  

5. Принимает отрицательные значения `id` и создаёт сущности с одинаковым идентификатором, как в пункте 1

6. Обрабатывает строковые значения `id` как серверную ошибку вместо ошибки клиента

7. Принимает дробные значения `id` и автоматически преобразует их в целые без уведомления клиента  

8. Не ограничивает длину поля `name`, позволяя передавать чрезмерно длинные строки  

9. Допускает использование специальных символов в поле `name` 

10. При создании питомца с уже существующим `id` сервер возвращает 200 и перезаписывает данные, что приводит к потере ранее имевшихся данных, это делает работу API непредсказуемой

11. Обрабатывает пустое тело запроса `{}` без ошибок, несмотря на отсутствие обязательных полей  

12. При передаче слишком большого значения `id` сервер возвращает ошибку 500 вместо ошибки валидации

### GET /pet/{petId}

1. При передаче строкового `id` возвращает 404 с внутренней ошибкой (`NumberFormatException`) вместо ожидаемой ошибки валидации 400, это указывает на некорректную обработку входных данных и раскрытие внутренней реализации, то же самое происходит и при передаче  слишком больших значений `id`  

2. Допускает неявное преобразование значений `id` (например, `09 → 9`), что может приводить к неоднозначной интерпретации


### GET /pet/findByStatus

1. Невалидный статус принимается без ошибки: значения не входящие в список `available`, `pending`, `sold` принимаются и возвращают 200

2. Обязательный параметр status не валидируется: при отсутствии параметра сервер возвращает 200 с пустым результатом вместо 400

3. Регистр статуса не нормализуется: `SOLD` и `sold` обрабатываются как разные значения, рекомендуется нормализовать значение status (например, приводить к нижнему регистру) перед валидацией или не принимать совсем

### GET /pet/findByTags

1. Если передано несколько тегов, то учитывается только первый: при передаче `?tags=tag1&tags=tag2` учитывается только первый тег, результат зависит от порядка параметров

2. Обязательный параметр tags не валидируется: при отсутствии параметра сервер возвращает 200 с пустым результатом вместо 400

### PUT /pet

1. Отсутствие валидации обязательных полей: метод принимает запрос без `name` и `photoUrls`,  поля удаляются из сущности, что приводит к потере данных и нарушению целостности объекта

2. При передаче id несуществующего питомца метод создаёт новую сущность вместо возврата 404

3. Отсутствие id обновляет дефолтную сущность: без передачи параметра `id` применяются изменения к дефолтному объекту вместо ошибки валидации. Поле id не обозначено как обязательное, однако используется для идентификации сущности при обновлении 

4. Непоследовательная валидация отрицательных id: `id=-98` принимается и обновляет дефолтную сущность, а `id=-08` возвращает 400, поведение для невалидных значений различается

### POST /pet/{petId}

1. Ответ не содержит обновлённые данные: после успешного обновления сервер возвращает неинформативный ответ без актуальных данных питомца

2. Невалидный petId возвращает 404 вместо 400: при передаче строки в `petId` возвращается 404 с `NumberFormatException` 400 (Invalid ID supplied), что указывает на некорректную обработку входных данных и раскрытие внутренней реализации

3. Возвращает 200 OK при отсутствии параметров `name` и `status` в запросе, однако изменения не применяются, не выполняется валидацию входных данных и не сообщается об ошибке, что делает поведение неочевидным

4. Принимает невалидное значение поля `status`, возвращает 200 OK и обновляет его, что указывает на отсутствие валидации входных данных, поведение такое же как и в `POST/pet`, для поля `name` аналогично


### POST /pet/{petId}/uploadImage

1. Загруженное изображение не привязывается к питомцу: после загрузки файла поле `photoUrls` питомца не обновляется, судьба загруженного файла неизвестна

2. petId не валидируется, параметр фиктивный: при передаче несуществующего `petId` сервер возвращает 200 и сохраняет файл, `petId` не используется в логике обработки

3. Поле file фактически обязательное вопреки спецификации: в спецификации `file` необязательное, однако его отсутствие при наличии `additionalMetadata` вызывает 500, реальное поведение противоречит спецификации

4. Тип файла не валидируется: метод принимает любой формат включая `.py`, `.txt` и другие "не изображения"

5. Файл 0 байт принимается без ошибки: пустой файл сохраняется как `./null, 0 bytes` вместо ошибки валидации

6. Превышение размера файла не обрабатывается: при загрузке файла 100 MB запрос завершается по таймауту, ограничение не задокументировано и не обрабатывается


### DELETE /pet/{petId}

1. Ведущие нули в petId обрабатываются без уведомления: `02` преобразуется в `2`, поведение идентично GET
2. Невалидный petId возвращает 404 вместо 400: cтрока `"abc"` и отрицательный `-1` возвращают 404 с `NumberFormatException` вместо 400, поведение идентично GET.
